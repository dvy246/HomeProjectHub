# HomePlanningHub: SEO & AdSense Audit Report

**Executive Summary:** This audit identifies key crawl gaps (such as robots.txt blocking compliance pages), metadata layout weaknesses (canonical prop passing bug causing trailing slash mismatch), 5 high-yield SEO search opportunities, and structured layout guidelines for re-integrating AdSense with zero layout shift.

---

## 1. Current SEO & Metadata Analysis

We reviewed the codebase and evaluated the current technical SEO setup:
- **Schemas (`src/lib/schema.ts`):** Extremely robust. The codebase leverages the `@graph` schema representation (`graphSchema()`) to combine multiple semantic schemas into a single injection:
  - `Organization` (global in `Layout.astro`)
  - `BreadcrumbList` (route breadcrumbs)
  - `WebApplication` (calculator classification)
  - `MathSolver` (formulas and interactive solving actions)
  - `HowTo` (step-by-step instructions matching the page text)
  - `FAQPage` (nested questions with matching `<details>`/`<summary>` markup)
  - `Article` (used on guides and cost comparisons, crediting DIY specialist Marcus Vance)
- **Layouts (`src/layouts/Layout.astro`):** Handles essential `<head>` details (meta viewport, social OG/Twitter tags, webfonts, Google Analytics, theme script). It preconnects/dns-prefetches Google AdSense (`pagead2.googlesyndication.com`) and Tag Manager, preparing the DOM for future ad loading.
- **Sitemap (`astro.config.mjs`):** Properly utilizes `@astrojs/sitemap`. It filters utility pages (e.g., `/saved/`, `/planner/`, `/projects/`) and customizes change frequencies/priorities (e.g., Homepage is 1.0, estimators and calculators are 0.8 weekly, compare matrix pages are 0.6 monthly).
- **Robots.txt (`public/robots.txt`):** Includes crawler rules allowing standard paths and disallowing utility paths. However, several critical gaps exist here.

## 2. Identified Crawl Gaps & Metadata Layout Weaknesses

### Crawl Gap 1: Robots.txt Disallowance of Compliance Pages
- **Observation:** `public/robots.txt` includes:
  ```txt
  Disallow: /privacy/
  Disallow: /terms/
  Disallow: /disclaimer/
  ```
- **Risk:** Google AdSense reviewers require access to a site's Privacy Policy, Terms of Service, and Disclaimer pages to verify compliance (cookie usage disclosures, liability, affiliate marketing). Blocking these pages in `robots.txt` prevents the AdSense validation crawler from accessing them, leading to immediate application disapproval (usually flagged as "Valuable Inventory: No Content" or "Site Behavior: Navigation").
- **Recommendation:** Remove these pages from the `Disallow` list in `public/robots.txt` and instead use `<meta name="robots" content="noindex, follow" />` on the individual pages via Layout's `noindex` prop if search indexing is not desired.

### Crawl Gap 2: Missing Allow Directives for Dynamic Directories
- **Observation:** Robots.txt explicitly allows:
  ```txt
  Allow: /calculators/
  Allow: /strategy/
  Allow: /quizzes/
  Allow: /guides/
  Allow: /compare/
  ```
- **Risk:** It does not explicitly allow `/estimate/`, `/playbook/`, or `/maintenance/` paths, though they are listed in the sitemap. While `/` is allowed, the inconsistent listing of explicit `Allow` directories can confuse search crawlers.
- **Recommendation:** Standardize robots.txt to either avoid listing folder-specific `Allow` rules (allowing `/` is sufficient) or ensure all public directories like `/estimate/`, `/playbook/`, and `/maintenance/` are explicitly allowed.

### Metadata Weakness 1: The Canonical Link Prop Passing Bug
- **Observation:** Across nearly all calculator and guide index pages (e.g., `src/pages/calculators/aluminum-weight/index.astro`, `concrete/slab.astro`, `paint/index.astro`), the frontmatter defines a trailing-slash canonical URL (e.g., `const canonical = "https://homeplanninghub.com/calculators/aluminum-weight/";`), but `<Layout>` is invoked without passing it:
  ```astro
  <Layout title={title} description={description} schema={schema}>
  ```
- **Impact:** `Layout.astro` falls back to the dynamic path:
  ```typescript
  const canonicalURL = canonical || new URL(Astro.url.pathname, Astro.site || "https://homeplanninghub.com").toString();
  ```
  If a user or crawler accesses a calculator without a trailing slash (e.g., `/calculators/concrete/slab`), the canonical tag will print `https://homeplanninghub.com/calculators/concrete/slab` (no trailing slash). However, the sitemap generated by `@astrojs/sitemap` enforces trailing slashes. This canonical mismatch creates indexing anomalies and duplicate content reporting in Google Search Console.
- **Recommendation:** Update all calculator pages to pass `canonical={canonical}` in their `<Layout>` parameters.

## 4. Future AdSense Placement & Core Web Vitals Guidelines

Re-enabling AdSense must not trigger Cumulative Layout Shift (CLS) or slow down user interactions. Below are guidelines for layout-friendly placements:

### A. Preventing CLS in Ad Container Wrapper
By default, ads load asynchronously, and empty ad boxes expand after loading, pushing page elements downward. To prevent this, we must define fixed height boundaries on the `AdSlot` wrapper component.

#### Proposed Layout Code for `src/components/AdSlot.astro`:
```astro
---
interface Props {
  slotId: string;
  format?: 'auto' | 'fluid' | 'rectangle' | 'horizontal' | 'vertical';
  responsive?: boolean;
  className?: string;
}

const { slotId, format = 'auto', responsive = true, className = "" } = Astro.props;
const clientID = import.meta.env.PUBLIC_ADSENSE_CLIENT_ID;

// Define specific height profiles to reserve layout space and prevent CLS
let minHeight = "min-h-[100px]"; // Fallback
if (slotId.includes("leaderboard") || slotId.includes("top")) {
  minHeight = "min-h-[90px] sm:min-h-[100px]"; // Top horizontal banners
} else if (slotId.includes("mid") || slotId.includes("content")) {
  minHeight = "min-h-[250px] sm:min-h-[280px]"; // Square block display ads
} else if (slotId.includes("sidebar")) {
  minHeight = "min-h-[600px]"; // Vertical tower ads
}
---

{clientID && (
  <div class={`ad-wrapper my-6 flex flex-col items-center justify-center border-y border-[var(--border)] bg-[var(--bg-inset)]/10 py-4 ${minHeight} no-print ${className}`}>
    <span class="text-[9px] uppercase tracking-wider text-[var(--fg-muted)] mb-2">Advertisement</span>
    <ins class="adsbygoogle"
         style="display:block"
         data-ad-client={clientID}
         data-ad-slot={slotId}
         data-ad-format={format}
         data-full-width-responsive={responsive ? "true" : "false"}></ins>
    <script is:inline>
      (adsbygoogle = window.adsbygoogle || []).push({});
    </script>
  </div>
)}
```

### B. Optimal Ad Unit Placements
1. **Top Leaderboard (`ad-top-leaderboard`):** Place directly below the `H1`/Intro paragraph and above the interactive calculator card. This captures early attention but does not block the primary user utility (the calculator itself).
2. **In-Content Mid (`ad-in-content-mid`):** Place between the calculator output and the text-heavy sections (e.g., below "Example Calculation" and above "Common Mistakes"). This breaks up long columns of text and matches natural user reading patterns.
3. **Sidebar Stickies (Recommended addition for Desktop):** For calculators that utilize a left-hand canvas and a right-hand sidebar (e.g., our SVG designers), if there is a vertical empty column layout on desktop, place a sticky vertical ad unit (`300x600` or `160x600`) in it.
4. **Anchor Ads (AdSense Auto Ads):** Enable bottom-screen anchor ads for mobile users. They float on screen and do not alter the page layout flow, guaranteeing 0% CLS.
